Agent Schema.

Beacon Analytics runs three AI agents — RevOps, Finance, and GTM — each computing MRR independently. They return three different numbers. Agent Schema solves it: one queryable schema holds every metric definition, dbt model lineage edge, and business glossary term. Every agent reads from the same place before acting. No hallucinated metrics. No conflicting answers.

What the demo does

The app opens with a side-by-side comparison: three agents all answering "What is our MRR?" with conflicting numbers, versus a single agent querying beacon_agents.metric_definitions first and returning the canonical answer. An interactive Schema Explorer — modeled after VS Code — lets the audience browse six live metric definitions (MRR, ARR, NRR, Churn Rate, CAC, LTV), four dbt models with lineage, and three Agent Schema tables with column-level documentation. The Agent Chat section replays four real agent questions with the three-step loop visible: query schema, receive definition, answer confidently. The pipeline diagram closes the technical story: sources → Fivetran → Iceberg → dbt Gold → Agent Schema → AI agents.

Why ODI is the story

Fivetran moves data from Stripe and Salesforce into Iceberg on Databricks. dbt transforms through three layers — staging, marts, and a dedicated agent schema layer. That final layer is an open standard co-developed by Fivetran and dbt Labs: every metric definition, model lineage edge, and business glossary term is published into queryable SQL tables automatically when a dbt run completes. Any agent — regardless of vendor or framework — can SELECT from the schema before acting. The context is governed by the same pipeline that built the underlying data. No bespoke prompt engineering. No drift between what the model says and what the warehouse contains.
