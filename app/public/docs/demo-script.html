Agent Schema — Demo Script

Total runtime: ~10 minutes. No login required. Works on any device. Open the demo before you start: fivetran-jasonchletsos.github.io/AgentSchema-ODI-Demo

Beat 01 Set the problem — the AI data trust crisis ~90 sec
Say this

"Every company I talk to right now is standing up AI agents. And the first thing that breaks isn't the model — the model is actually pretty good. What breaks is the numbers. You have a RevOps agent answering questions about MRR, a Finance agent doing the same thing, a GTM agent doing the same thing — and they all return different answers. Because they each brought their own definition of what MRR means."

"This isn't an AI problem. It's a data governance problem. And Fivetran and dbt Labs built something to solve it. It's called Agent Schema — and it's an open standard for giving every agent a single place to read from before it acts."

If you're in front of a technical audience, add: "It's literally a schema in your warehouse — beacon_agents — three tables: metric definitions, model lineage, business glossary. Any agent can SELECT from it. No proprietary SDK. No new service to manage."
Beat 02 Problem section — show the conflict ~60 sec
On screen: scroll to the Problem section (second section of the page)
  • Point at the left panel — three agents, three different MRR answers
  • Point at the right panel — one agent querying the schema, one correct answer
Say this

"Here's what the 'before' looks like at Beacon Analytics — a fictional B2B SaaS company I use for this demo. Three AI agents all asked the same question: 'What is our MRR?' Three different answers. Each agent pulled from somewhere slightly different. None of them were technically wrong — they just weren't using the same definition."

"Now look at the right side. Same question, but this time the agent queries the Agent Schema first — then it answers. One query. One definition. Consistent every time."

If someone asks about the fictional company: "Beacon Analytics is made up — all the numbers are synthetic. The architecture is real. Same pattern works with Stripe and Salesforce regardless of company."
Beat 03 Schema Explorer — browse the definitions ~3 min
On screen: scroll to the Schema Explorer section

"This is the Agent Schema Explorer — think of it as VS Code for your metric definitions. Three tabs: Metrics, Models, Tables. I'll start with MRR since that's the one we just talked about."

  • Show the MRR entry selected in the sidebar — note the plain-language definition
  • Point at the Formula line
  • Scroll down to the SQL Reference block — walk through the syntax highlighting
  • Point at the Metadata band: Owner (Maya Chen), Team (RevOps), Refresh (Daily at 06:00 ET)

"Every metric has an owner, a team, a refresh cadence, and a canonical SQL query. When the RevOps agent asks about MRR, it gets this definition — not its own interpretation of a Stripe table."

Switch to Models tab
  • Click fct_mrr_movements — show the column table
  • Point at the Lineage section: upstream from stg_stripe_subscriptions and stg_salesforce_accounts

"The model lineage is also in the schema. An agent can see exactly what feeds this model before it writes a query. It's not reading docs. It's reading SQL tables — the same ones that Fivetran and dbt build automatically."

Switch to Tables tab
  • Click beacon_agents.metric_definitions — show row count (47 metrics) and column list
  • Point at the auto-publish note at the bottom

"This is the actual table the agents query. 47 metric definitions. Published automatically via GitHub Actions every time a dbt model or metric definition changes. No manual updates. The agent always gets the current version."

Beat 04 Agent Chat — show the loop live ~2 min
On screen: scroll to the Agent Chat section

"Now let me show you what the agent actually sees. I'll pick a question — let's use MRR."

  • Click "What is our MRR this month?"
  • Watch the three-step animation: spinning dot (querying schema) → green checkmark (schema result) → answer bubble

"Three steps. Step one: the agent fires a query against the Agent Schema — you can see the exact SQL. Step two: the schema returns the canonical definition, formula, and source model. Step three: the agent uses that definition to answer — and it cites the source. No hallucination. No inventing a formula. It reads, then it answers."

  • Click "How is Net Revenue Retention calculated?" — point at the formula in the schema result

"NRR is a great example. The denominator is starting MRR on the first day of the period — not the average. That distinction matters. Finance owns this definition. Without the schema, every agent makes its own call on which denominator to use. With it, they all use Finance's."

Optional: click the other two questions if the audience wants to see more. "What sources feed the revenue model?" and "Which team owns the churn metric?" both demonstrate the lineage and ownership use cases.
Beat 05 Pipeline — the technical story ~90 sec
On screen: scroll to the Pipeline section

"Here's the full architecture. Six nodes. Fivetran moves data from Stripe and Salesforce into Iceberg on Databricks. dbt runs three transformation layers — staging, mart, and the agent schema layer. That last node — Agent Schema — is the one we just browsed. It publishes every metric definition, lineage edge, and glossary term into queryable tables. Then any AI agent reads from it before acting."

  • Point at the Agent Schema node — glowing cyan, "OPEN STANDARD" badge
  • Point at the three feature cards below: dbt Wizard, dbt State, Great Expectations

"Below the pipeline you can see three capabilities that make the schema trustworthy. dbt Wizard generates the agent schema layer automatically. dbt State means only changed models republish — incremental by default. Great Expectations validates the data before it ever reaches the schema. Every agent reads from a tested, governed, up-to-date source."

If you're in front of a data engineering audience: "The dbt State integration means you're not re-running 200 models to update one metric definition. Only the models that changed run. The agent schema stays current without burning your warehouse credits."
Beat 06 Close — what to leave them with ~30 sec

"Agent Schema isn't a new service. It's a new layer in the pipeline you're already building. Fivetran moves the data. dbt transforms and publishes it. Your agents query before they act. The whole thing is open format — standard SQL, standard Iceberg, no vendor lock-in. And it ships in the same pipeline run that built the underlying data."

"If you're already using Fivetran and dbt, you're one pipeline step away from this."

Q&A Common objections and responses
"We're not using dbt. Does this still work?"
"Agent Schema is an open standard — it's a schema in your warehouse, not a dbt feature. dbt Labs is the co-author, which means dbt makes publishing automatic. But you can populate the tables with any transformation layer. The spec is open."
"How do agents discover that the schema exists?"
"You add it to the agent's system prompt or tool definition once: 'Before answering any metric question, query beacon_agents.metric_definitions.' After that, every agent in your org uses it automatically. One line per agent. Never update it again."
"What if a metric definition changes? Does the agent pick it up automatically?"
"Yes. The schema is just a SQL table. The next time the agent runs, it reads the current row. No prompt updates. No redeployment. The governance is in the pipeline, not in the agent."
"Is this production-ready or a roadmap item?"
"The open standard is live — the spec is published and Fivetran has reference implementations running. The demo app you're looking at is built on the real schema structure. Talk to your Fivetran rep about implementation timelines for your stack."
"Can this work with Claude / GPT / LangChain / our internal model?"
"Any agent that can issue a SQL query can use Agent Schema. It's not a proprietary API. The agent queries a table, reads a row, uses the definition. That pattern works with any framework."
